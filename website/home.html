Software Design and Architecture

This is the online textbook for the Software Design and Architecture module run by Manchester Metropolitan University as part of a Bsc Software Engineering programme.

Pre-requisites

The examples are in Java, so you will need Java programming knowledge to understand the examples. You should be comfortable with Java syntax and concepts such as classes, objects, methods, inheritance, and interfaces.

All examples are implemented as simple command-line applications, rather than using any GUI or Web frameworks. This because we want to concentrate on design, without having to spend time on the technicalities of a particular UI or Web framework.

Terminology

We cannot discuss software design without a common understanding of the terminology. When we introduce terms (special words or names that are common in the field), they will be in bold.

Java language keywords will be formatted like this: keywords.

There is no single standards body or agency the set or defines these terms precisely (sorry) therefore you might find different names in books and papers that mean the same thing. We have tried to include some alternative names in the material.

  • For generic software design and modelling terms our reference for naming is the Unified Modeling Language Specification (Object Management Group, 2017)

  • For Java specific terminology our reference is the Java® Language Specification (Oracle, 2025).

  • Where there is a conflict between UK English and US English spelling, we will use the US version as US English is most common in the software engineering.

Principles and Patterns

We might wish that designing and writing software was just a matter of following a set of rules. Instead, software design deals with principles and patterns.

A Principle in software engineering is a guideline or a best practice. Understanding and being able to apply the principles will help you make better choices in your software design by following the principle. We are covering the SOLID principles written up in a series of books such as Martin and Martin (2007).

A Design Pattern is more practical than a principle. Design patterns are general, reusable solutions to common problems encountered during software design. The patterns we cover here were originally defined in Gamma at al, (1994).

Design Patterns provide a template or blueprint for solving common design problems. They are expressed in diagrams and code and provide a starting point for adaptation to your specific problem.

Advice

We have included some advice in the material. It is our opinion but is not definitive - every coding and design problem has its own context and forces driving the design, the unique combination of which determines the best design.
